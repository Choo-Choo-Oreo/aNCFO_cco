{{Version|1.12}}
Within Hearts of Iron IV, there are several types of data structures, used to store data to be used for later. This can be used in-game as a way to check if some effect block was executed or not, as a counter of sorts, or something else.

__TOC__

== Constants ==
Constants serve as a way to use the same value with a reference. The constants are marked with the <code>@</code> symbol when defining or using them. An example definition is the following:<pre>@CONSTANT_1 = 10
@CONSTANT_2 = "TAG_my_idea"</pre> This can later be used in the file as, for example, <code>cost = @CONSTANT_1</code>. These definitions can be in any point of the file.

Constants are available in the vast majority of game files, including most, if not all, *.txt and *.gui files. Constants can be used as a way to link several values to be the same if they can easily be changed at any point in the development. However, constants that store a string for its value can only be used in triggers and the attributes of elements and windows, trying to use them in an effect will result in an "invalid database object for effect/trigger" error.<br/>
For example, if there is a large decision system where every decision is intended to have the same cost, then these constants may be used if the political power cost gets changed mid-development for better balance.<br/>
Another example would be a scripted GUI container: if a large multitude of elements are to be at the same X or Y position, it might be worth it to link them to be a constant in case their position could be changed for better appearance or to fit in another GUI element.

Example [[Decision modding|decision file]] that utilises constants:
<pre>@CONSTANT_1 = 10
@CONSTANT_2 = "TAG_my_idea"
TAG_decision_category = {
    TAG_decision = {
        icon = test_icon
        cost = @CONSTANT_1
        complete_effect = {
            if = {
                limit = {
                    has_ideas = @CONSTANT_2
                }
                add_stability = 0.1
            }
        }
    }
}</pre><noinclude>

=== Defines ===
{{Main|Defines}}
Defines are a particular type of constant that are used within internal calculations only, defined within {{path|common/defines/*.lua}} files. These cannot be referenced in any other files and are only used within the internal, unchangeable code.<br/>

Some of the things included in the defines are that events by default stay for 13 days before auto-selecting the first option<ref><code>NDefines.NGame.EVENT_TIMEOUT_DEFAULT = 13</code> in [[Defines]]</ref> or that the ai_will_do value is 50% higher for focuses which were unlocked by the just-completed focus<ref><code>NDefines.NAI.FOCUS_TREE_CONTINUE_FACTOR = 1.5</code> in [[Defines]]</ref>. Most defines are numerical in nature, but some can be strings or blocks, such as the end date at which the game over screen appears<ref><code>NDefines.NGame.END_DATE = "1949.1.1.1"</code> in [[Defines]]</ref> or the list of experience need for a division to progress to the next level<ref><code>NDefines.NMilitary.UNIT_EXP_LEVELS = { 0.1, 0.3, 0.75, 0.9 }</code> in [[Defines]]</ref>.<br/>
The list of defines is vast and touches on every mechanic in the game, so if something can't be changed within the mechanic's respective files, it may be worth to check defines. Defines don't include the [[Modifiers#Static modifiers]], which can serve a similar purpose but with modifier blocks.

As defines are Lua code rather than a PDXscript-interpreted file, any Lua code can go in there, however modules allowing to go out of this directory are not available. As a consequence of this, '''there is no need to copy the entire file to edit it'''. Defines, including graphical defines, are merely a Lua-contained array, and it is possible to modify a single member of the array in Lua using, for example, <code>NDefines.NGame.START_DATE = "1936.1.2.12"</code>. Each of these lines is contained on a separate line and {{hover|They are present in the base game defines file due to the fact that NDefines is one single large array, and when defining an array with members, the members are separated with commas. However, separate commands are not separated with commas.|there are no commas separating them}}, such as the following:

<SyntaxHighlight lang="lua">
NDefines.NGame.START_DATE = "1936.1.2.12"
NDefines.NGraphics.COUNTRY_FLAG_TEX_MAX_SIZE = 2048
</SyntaxHighlight>

This goes into a separate file set to be evaluated after the base game defines, which is done by making the filename be later in the alphabetical order (decided by Unicode character IDs). As most Unicode characters (including every uppercase or lowercase letter) have larger IDs than the code used for the number 0, the filename is almost irrelevant.<br/>
In the base game files, the graphical defines are set to be merged into NDefines in the last line of the base game file. However, since Lua does not create copies of tables by default, the base game's NDefines contains pointers to the actual elements, which are contained within NDefines_Graphics regardless. For this reason, modifying either NDefines or NDefines_Graphics works for changing graphical defines.<br/>
A mod should '''never''' contain the 00_defines.lua and 00_graphics.lua files within of itself: these files are commonly changed even in minor game updates, and having a define missing from a file results in the mod being unstable, potentially having a crash on startup.</noinclude>

== Flags ==
In this case, flags are used in the sense of a computer variable that is used to set data to and later read it from, commonly but not necessarily of a boolean value (i.e. either 0 or 1). For editing the flags used for representing countries in-game, see [[Country creation]] for the starting flags and [[Cosmetic tag modding|cosmetic tags]] for changing them in the middle of the game.

Flags within PDXScript are a 32-bit signed integer, ranging from -2147483648 to 2147483647. A flag can only be set to a specific value (Potentially for a certain amount of days), modified in an additive way by a certain value and read, no other operations can be done. [[#Variables|Variables]] serve as a more operational alternative, with far more capacity for changing (Such as multiplication or non-integer values, reading game values, being able to set to other database entries rather than numbers, etc). Flags are still recommended to use as a boolean value, as these are more optimised than the variables, and so using variables will lead to an unnecessary increase in memory usage and will lead to the game running slower compared to flags.

Flags are divided into 5 scopes:
* Global. These can be read from anywhere in the game, without the need for scoping.
* Country. These are assigned to individual countries, being impossible to assign or read elsewhere. A country flag can have different values depending on which country it's checked in.
* State. These are assigned to individual states, being impossible to assign or read elsewhere. A state flag can have different values depending on which state it's checked in.
* Character. These are assigned to individual characters, being impossible to assign or read elsewhere. A character flag can have different values depending on which character it's checked in.
* [[Military industrial organisation]]. The <code>mio</code> shortnening is used in code for them, such as <code>set_mio_flag</code>.
In general, the same principles apply to them with little changes. While unit leader flags exist, these are marked as deprecated and are likely to be removed within the next updates.

A flag can be set in ''any'' [[effect]] block. There is no file where flags need to be defined, they will exist if set and stop existing when cleared.<br/>
The simplest formatting for setting a flag is <code>set_country_flag = my_flag_name</code>. This will create the country flag (If it did not exist otherwise) and set it to be a value of 1. ''This has no tooltip'' and so it will be invisible to the player. [[Effect#custom_effect_tooltip|custom_effect_tooltip]] may be used to tell the player that something will be done, if wanting to keep them in the know.<br/>
Afterwards, that flag can be checked with <code>has_country_flag = my_flag_name</code> as a trigger. This is set to be true if the flag was set, regardless of which value it has. This will create a tooltip using the flag's name as the [[localisation|localisation key]]. As such, <code>my_flag_name: "Has my flag"</code> will make that appear when the country flag is checked.<br/>
If a flag is no longer useful, it can be cleared by using <code>clr_country_flag = my_flag_name</code>, which'll mark it as no longer being set. This also has no tooltip and the player will not be aware unless specifically told with the custom effect tooltip that the flag was cleared.

The exact same principles apply with global, state, and character flags. The only differences are in the used effects/triggers (In which case the word <code>country</code> is swapped with the type's name like <code>set_global_flag</code>, <code>has_state_flag</code>, or <code>clr_character_flag</code>) scopes in which they can be set and where they can be checked. For global flags, it can be done in any scope rather than the 'global' scope: it can be set in either country, state, character, or rare other scopes.

Flags can also be targeted towards a certain country. This is usually done by appending the <code>@TAG</code> to the name of the flag, though it can be placed anywhere within the tag name except for the first character. For example, <code>set_country_flag = my_targeted_flag_@PREV</code> will set it to be targeted towards the country represented by PREV, which we'll take as TAG. ''This just appends the PREV's tag to the flag's name''. After this, <code>has_country_flag = my_targeted_flag_TAG</code> will come up as true for the same country it was set in. This can be used to create dynamic flags without relying on [[Effect#Meta effects|meta effects]]. You cannot have more than one target specifier in a single tag, You can only use @ROOT, @PREV, @FROM, and @THIS, even @PREV.PREV does not work. Both of these restrictions do not apply to targeted variables, so those may be a more flexible alternative if you need.

=== Example ===
The most common use of flags is to simply to mark a certain event (in the generic sense of a word, not necessarily the in-game [[events|event mechanic]]) as having occurred, such as an election, an invasion, a diplomatic deal, et cetera. For example, here's an event that starts off an election campaign in the country:
<pre>country_event = {
    id = algeria_election.0         # Namespace is assumed to exist.
    title = algeria_election.0.t
    desc = algeria_election.0.desc
    
    is_triggered_only = yes
    
    immediate = {
        set_country_flag = ALG_1936_elect_campaign  # Since set_country_flag has no tooltip, no need to hide it.
    }
    
    option = {
        < ... >                     # Irrelevant what's in the option
    }
}</pre>Flags provide a way to mark that this event has been received, in this case set within the immediate effect block of the event.<br/>
In order to utilise this, the <code>has_country_flag</code> trigger will have to be used similarly to this example decision:
<pre>ALG_1936_campaign_category = {         # Decision category is assumed to exist and contain 'allowed = { tag = ALG }'
    ALG_1936_select_candidate_A = {
        available = {
            has_country_flag = ALG_1936_elect_campaign
        }
        
        icon = ALG_elect
        cost = 50
        fire_only_once = yes
        
        complete_effect = {
            country_event = algeria_election.1
        }
    }
}</pre>
This will work fine, however, as is, the player will see either a {{icon|yes}} check mark or a {{icon|no}} cross next to 'ALG_1936_elect_campaign', with no indication what that truly means. In order to correct that, a localisation entry can be added to any working localisation file (i.e. is an *.yml file within the {{path|localisation/}} folder or any of its subfolders, has the internal name of the language in the end of filename before the extension, and is encoded in UTF-8 with the byte order mark present):
<pre> ALG_1936_elect_campaign: "The 1936 election campaign is currently ongoing"</pre>
In order to clear the flag when needed, the <code>clr_country_flag</code> effect would be used as such:
<pre>country_event = {
    id = algeria_election.100         # Namespace is assumed to exist.
    title = algeria_election.100.t
    desc = algeria_election.100.desc
    
    is_triggered_only = yes
    option = {
        name = algeria_election.100.a
        trigger = {
            has_country_flag = ALG_candidate_A_won
        }
        clr_country_flag = ALG_1936_elect_campaign
        < ... >                     # Irrelevant what else is in the option
    }
    option = {
        name = algeria_election.100.b
        trigger = {
            has_country_flag = ALG_candidate_B_won
        }
        clr_country_flag = ALG_1936_elect_campaign
        < ... >                     # Irrelevant what else is in the option
    }
    < ... >                         # Other potential options
}</pre>
Note that neither of the effects that modify country flags have tooltips, so [[Effect#Effect tooltips|a custom effect tooltip can be used to notify the player]] in some cases.<br/>
Everything is the exact same for flags of other scope types.
=== Non-boolean usage ===
This will cover the non-boolean usage of flags. [[#Variables|Variables]] usually do this job better, but this still has some usages.

The <code>set_<type>_flag</code> effect can also be expanded with the optional <code>days</code> and <code>value</code> arguments. This will appear like the following:
<pre>set_state_flag = {
    flag = my_state_flag
    days = 50
    value = 3
}</pre>This in particular will set the flag my_state_flag to the state to be equal to 3 for 50 days. After these 50 days end, the flag will be cleared completely, reset to 0. If days are left out, the flag will be set to that value forever. If the value is left out, the flag will not be changed.
And the trigger if a flag is set can also be expanded as such:
<pre>has_global_flag = {
    flag = my_global_flag
    value > 2
    days > 30
    date < "1938.1.1"
}</pre>While the flag and the value arguments are self-explanatory, <code>days</code> checks how many days it was ''since the flag was set'' (Not how many are remaining), while the <code>date</code> checks the date at which the flag is set. Every argument aside from the <code>flag</code> one is optional, just assumed to be always true if not set. This trigger will still always fail if the flag was never set or if it was cleared.<br/>
Additionally, it is possible to ''modify'' a flag as such:
<pre>modify_character_flag = {
    flag = my_character_flag
    value = 4
}</pre>This ''adds'' the amount to the flag, with negatives being possible. However, there's one catch: '''this does nothing if the flag is not set''', having a value of 0.

Each of the preceding examples can be used for any flag type.

== Event targets ==
Event targets serve as a primitive way to set a pointer to a scope (country, state, character) to be referred to later. Any single event target can only point towards a single scope at a time, however a scope may have several event targets at the same time. Similarly to flags, there is no set folder for event targets, they can be assigned in any event block.<br/>
There are two types of event targets: regular and global. The difference between them is purely when they get cleared.

A scope can be checked if it has an event target assigned using <code>has_event_target = name_of_event_target</code>.
=== Global event targets ===
Global event targets can be declared using <code>save_global_event_target_as = my_event_target</code> as an effect. This will create the global event target with the name of <code>my_event_target</code> and it will last forever. Afterwards, it can be referred to as <code>event_target:my_event_target</code>, both as a scope and as a target.

If <code>save_global_event_target_as = my_event_target</code> is used when the event target is already set, the event target will change scopes to the one where the effect is used.<br/>
Additionally, <code>clear_global_event_target = my_event_target</code> (to be used in any scope, not necessarily the event target) will clear the specified global event target entirely, while <code>clear_global_event_targets = ROOT</code> will clear every global event target from the specified scope.

For example, by using [[Effect#start_civil_war|start_civil_war]] as an effect block, the following will assign the TAG_democratic_side event target to the revolter:
<pre>start_civil_war = {
    size = 0.5
    ideology = democratic
    save_global_event_target_as = TAG_democratic_side
}</pre>
After this, TAG_democratic_side event target will refer to the democratic side. For example, this code later in another effect block can be used to transfer a state and a part of the armies to the event target:
<pre>event_target:TAG_democratic_side = {
    transfer_state = 123
}
transfer_units_fraction = {
    target = event_target:TAG_democratic_side
    size = 0.2
}</pre>
It would make sense to clear the event target after it no longer exist, such as by using the [[On_actions#on_civil_war_end|on_civil_war_end on action]].

When using the event target as a scope for a namespace in localisation, the <code>event_target:</code> prefix is not needed, instead it'll look like <code>[TAG_democratic_side.GetName]</code>.
=== Regular event targets ===
Regular event targets can be declared using <code>save_event_target_as = my_event_target</code> as an effect. The difference between regular and global event targets is in when they get cleared. While global event targets are used forever and can only be cleared manually, regular event targets only last as long as the effect block does (carrying over into events fired within as well), clearing automatically afterwards, and there is no way to clear them manually.

For example, this in an effect block will save a random owned and controlled state as the event target <code>target_state</code> and then fire the event mod_event.0 in 10 days:
<pre>random_owned_controlled_state = {
    save_event_target_as = target_state
}
country_event = { id = mod_event.0 days = 10 }</pre>
Assuming no other events, this event target will only exist within this effect block and the event mod_event.0 (Where it can be used in its corresponding trigger or effect blocks). If the event mod_event.0 fires another event within a corresponding effect block, this event target will exist in that event as well, meaning that this carries over to the entire mod_event.0-starting event chain. For example, the generic event [[Generic events#generic.23|generic.23 ([From.GetNameDefCap] Demands Control of [demanded_prov_target.GetName])]] uses this with demanded_prov_target as the event target. In order to make ROOT demand the state 123 from TAG using this, this effect block will be used:
<pre>123 = {
    save_event_target_as = demanded_prov_target
}
TAG = {
    country_event = generic.23
}</pre>

Event targets can be used in ''most'' effect blocks, they are called event targets only because they carry over to events. However, there are a few effect blocks, such as scripted diplomatic actions, where event targets are impossible to set.
== Country tag aliases ==
Country tag aliases are somewhat similar to event targets: they are used as pointers towards a specific scope, a single alias can point towards a single scope, and a single scope can have several aliases. However, there are differences. Primarily, country tag aliases can only point towards countries (as per the name) and they are set and cleared entirely automatically without any manual involvement. A country tag alias can point towards a non-existing country.

Country tag aliases are stored within {{path|common/country_tag_aliases/*.txt}} files. Each country tag alias is stored as a separate block within the file, with the name of the block being the name of the alias. Additionally, the country tag aliases serve as a trigger block, so any trigger can be used within. These are the arguments that can be used within:
* <code>variable = global.var_name</code> - This will make sure that the country tag alias is equivalent to the country that the specified [[#Variables|variable]] points to, in this case <code>var_name</code> assigned to the global scope.
* <code>event_target = my_event_target</code> - This will make sure that the country tag alias is equivalent to the country that the specified '''regular''' event target points to, in this case <code>my_event_target</code>.
* <code>global_event_target = my_event_target</code> - This will make sure that the country tag alias is equivalent to the country that the specified '''global''' event target points to, in this case <code>my_event_target</code>.
* <code>original_tag = TAG</code> - This puts every country that originates from TAG (created with [[Effect#start_civil_war|start_civil_war]] or [[Effect#create_dynamic_country|create_dynamic_country]]) as a possible target for the alias. In this case, further specification is needed with triggers. This is equivalent to the [[Conditions#original_tag|original_tag trigger]], but this is treated as an argument here as it does make certain errors go away.
* <code>targets = { ABC DEF GHI }</code> - This is a whitespace-separated list of country tags that the country tag alias can select. In this case, further specification is needed with triggers.
* <code>target_array = global.my_array</code> - This puts every country that's within the specified array (<code>my_array</code> in the global scope in this case) as a possible target for the country tag alias. In this case, further specification is needed with triggers.
* <code>country_score = { ... }</code> - This is a [[AI modding#MTTH blocks|MTTH block]] evaluated for each country. The country which has the highest total score will be picked for the country tag alias, provided the conditions imposed by every other argument are met.
* <code>fallback = TAG</code> - This is what the country tag alias will point to if no country fulfills the conditions specified within triggers or other arguments.

Here are some examples of country tag aliases:
<pre>MOD = {
    targets = { ABC DEF GHI }
    has_country_flag = my_flag
    country_score = {
        base = 1
        modifier = {
            add = 2
            tag = ABC
        }
        modifier = {
            add = 1
            tag = DEF
        }
    }
    fallback = ABC  # This will point to ABC, DEF, or GHI if they have the specified country flags. ABC is prioritised over DEF, which is prioritised over GHI. If neither have the country flag, defaults to ABC.
}
ARB = {
    target_array = global.arab_countries
    has_cosmetic_tag = ARABIA_UNITED    # This will point to any country in the global.arab_countries array if they have the ARABIA_UNITED cosmetic tag, provided they exist.
    exists = yes
}</pre>
This is most commonly used with [[Effect#start_civil_war|start_civil_war]] or [[Effect#create_dynamic_country|create_dynamic_country]]. For example:
<pre>SDA = {
    original_tag = SDI
    has_country_flag = SDI_side_A
}</pre>
In order for this country tag alias to get used for side A of the civil war, it must have the country flag set. This is usually done directly in the effect that spawns it:
<pre>start_civil_war = {
    ideology = neutrality
    size = 0.5
    set_country_flag = SDI_side_A
}</pre> After doing this, it's possible to use <code>SDA</code> as if it's a regular country tag, e.g. <code>SDA = { add_political_power = 100 }</code> or <code>transfer_state_to = SDA</code>.

== Variables ==
A variable within Hearts of Iron IV is a signed 32-bit fixed point number with scaling of <math>\frac{1}{1000}</math>. As such, it ranges from -2 147 483.648 to 2 147 483.647. Adding further amounts above or below the variable will result in an [[wp:Integer overflow|overflow]], causing the value go from the maximum to the minimum or vice-versa. 

Variables are not stored within any particular file and can be created dynamically with any [[effect]] or (temporary variables-only) [[trigger]]. Each database object (such as countries, states, ideas, or even equipment types) has a unique internal numeric ID, which can be assigned to a variable. For example, setting a variable's value to a country tag will make it take on the ID of the tag, and then the variable will be taken as a country when used as a target or as a scope. For [[#Token-valued variables|tokenizable database types]], a <code>token:</code> prefix is necessary when assigning a literal, for example: <code>var_name = token:some_idea_name</code>.

When using a variable as a scope, the <code>var:</code> prefix is needed, however in some cases the prefix is optional: most notably when using it as the target of an effect or trigger. For example, in <code>transfer_state = var:my_state_variable</code>, the <code>var:</code> can be omitted with no difference (For some cases, there will be logged errors, yet it will work in-game), but in <code>var:my_country_variable = { transfer_state = 123 }</code> the <code>var:</code> is ''mandatory''. Despite functioning similarly to a scope, the <code>var:</code> prefix is optional when a variable is used as a weight inside of [[Effect#Random effects|random_list]]. Same as with event targets, using variables in localisation for namespaces does not require the <code>var:</code> prefix, such as <code>[?var_name.GetName]</code>. A good rule of thumb is that when passing a variable in a place where a raw token (rather than, say, a quoted string, a number, or a numeric state ID) is acceptable, it must usually be prefixed with <code>var:</code> so that the parser can distinguish whether the token being passed is meant to reference a database object or a variable. If <code>var:</code> is absent in a location where it should be present, the code may work, but the game will often complain in the error log. In general, it is best to err on the side of caution in this regard, as what may result in a merely-complaining error in one context, might result in an effect-nullifying, or worse, trigger-falsifying, error in another context. However, <code>var:</code> is always optional in variable and array manipulation operators (including <code>check_variable</code>), and so it can be elided in those operators for readability's sake.

Within their names, variables can have any word character, i.e. the 26 English letters (Either lowercase or uppercase), underscores, and the decimal digits from 0 to 9. However, a variable name cannot ''begin'' with a decimal digit, so a name like <code>123_my_variable</code> is invalid. This requirement common in many programming languages, and is implemented here for the same reason that the <code>var:</code> prefix exists: so that the parser can determine what syntactic type a token will be from the first (few) characters, and therefore doesn't have to worry about backtracking as much. In addition, it is recommended to avoid any names by variables that are already used for something else. For example, naming a variable <code>capital</code> can prevent any subsequent usage of the [[#Game variables|game variable with the same name]] within the internal code. A common naming convention to avoid overlap is to add a prefix signifying where it's used, such as the country's tag (<code>POL_capital</code>) or the name of the mechanic (Usually shortened like <code>PC_capital</code>).

=== Variable types ===
There are two types of variables: regular and temporary. There are the following differences between the two:
* Different effects are used to set and modify regular and temporary variables. Typically, the difference is just in changing <code>variable</code> to <code>temp_variable</code>.
* Temporary variables only exist within the effect or trigger block where they're defined, similarly to regular event targets (although without carrying over to events). This does not mean they don't survive outside of the scope where they're defined, but the '''entire'' effect block, such as a focus reward or an event option. Meanwhile, regular variables exist forever. Note that this, however, does not always function properly in regards to scripted effects or triggers: temporary variables defined within of them may not still exist outside of them within the same effect/trigger block. However, a temporary variable defined before the scripted effect or trigger's usage will always carry on into it and get modified properly within.
* Due to the previous point, '''temporary variables can be defined and modified in trigger blocks''' as well as in effect blocks. This enables them to be computed as-needed in scripted triggers called from the <code>trigger</code> block of scripted localizations. Meanwhile, regular variables are restricted to only being defined in effect blocks.
* Temporary variables are unscoped: they will remain the same regardless of in which scope they're defined and where they're checked. Meanwhile, regular variables are restricted to scopes, so that the variable has different values depending on which scope it's checked in, such as how different countries may have different stability values.

=== Scoping ===
'''Scoping only exists for regular variables.''' They can and must be assigned to countries, states, characters, and to the global scope.<br/>
If nothing else is specified, the current scope, or [[Scopes#THIS|THIS]], is used as the scope of the variable. For example, <code>ABC = { set_variable = { my_var = 1 } }</code> will set the variable with the name of my_var within the scope of ABC. <code>CBA = { check_variable = { my_var = 1 } }</code> would then come up as false, since it checks for the variable only within CBA, while the variable is set in ABC.

The scope where the variable is read can be changed by prepending any [[Scopes#Dual scopes|dual scope]] that can be used as a target and can be used in any scope (For example, character scopes can not be prepended before variable because character scopes are not any-scoping scopes, which means they can not be used in other character scopes.) before the variable's name, separated with a dot, such as <code>PREV.PREV.var_name</code> being the value of <code>var_name</code> as checked for <code>PREV.PREV</code> or <code>event_target:my_event_target.var_name</code>. This also includes <code>global</code> in order to assign it globally, '''which is necessary to scope into global'''. Other variables, if pointing to a scope, can also be used, but the <code>var:</code> is unneeded. Variables must be chained using the <code>:</code> separator (rather than the usual dot), such as <code>ABC.capital:var_name</code> being the variable <code>var_name</code> as it's assigned to the capital state of the country ABC, where <code>capital</code> is a built-in variable pointing to the capital state of the country. Note that the scope specifier preceding <code>capital</code> is separated from it by a <code>.</code> since it is a static (that is, non-variable) scope. The way to intuit this syntax is that the whole <code>ABC.capital</code> clause forms a variable scope specifier, in which <code>var_name</code> is accessed.

Here are some examples of scoped variable references:
<pre>
var_name                                    # Scopes to THIS
GER.var_name                                # Scopes to GER
123.var_name                                # Scopes to state 123
global.var_name                             # Scopes to global
province_controllers^1234:capital:var_name  # Scopes to the capital state of the country that controls the province 1234.
</pre>
In practice, scoping to distinguish variables may look like this:
<pre>every_country = {
    set_variable = { weighted_manpower = 0 } # Reset the counter to default in case it already exists
    every_owned_state = {
        set_variable = { weighted_manpower = state_population_k } # Completely unrelated to the above variable due to being in a different scope
        if = {
            limit = {
                NOT = { is_core_of = PREV } # Is core of the owner
            }
            divide_variable = { weighted_manpower  = 10 } # Divides the state's equivalent
        }
        add_to_variable = { PREV.weighted_manpower = weighted_manpower } # Adds to the owner's counter
    }
}</pre>In this case, each state has a <code>weighted_manpower</code> variable which is equal to the population in thousands if it's a core of the owner at the time of being set and it's divided by 10 otherwise. Each country will also have a <code>weighted_manpower</code> variable, which sums up the weighted manpower of each state. Despite having the same name, these variables are completely unrelated to each other due to one being country-level and the other being state-level.

=== Trigger usage ===
'''Only temporary variables can be modified within a trigger block.''' Regular variables can only be used in effects, while temporary can be used in both triggers and effects.

However, there's one aspect of triggers that's important to know: ''when'' each scope stops evaluating. If a scope stops evaluating, then the variable will not be modified anymore. In general, [[Scopes#Trigger scopes|trigger scopes]] that can select multiple targets are divided into two groups: <code>all_<scope type></code> and <code>any_<scope type></code>. The first group, requiring all targets to meet triggers, evaluates each target in order and stops when it encounters one where the triggers aren't met. The second group, requiring any target to meet triggers, evaluates each target and stops when it encounters a true one.<br/>
Empty trigger blocks are considered true by default, so one would instantly stop the validation of the second group (<code>any_</code> triggers). For this reason, [[Scopes#Scope limits|since scope limits are not supported in trigger scopes]], then the proper usage for modifying temp variables would be an [[Conditions#if|if statement]] that modifies the variable if the limit is met with no other triggers that can come up as true or false coming up.

=== Localisation ===
{{Hatnote|Parts of this section are transcluded from {{Section link|Localisation|Formatting variables}}}}
In order to display the value of a variable in localisation, <code>[?var_name]</code> is used, with the question mark signifying that it's a variable. '''The question mark is mandatory''' in order to distinguish the variable from the event targets. For example, such a localisation value will work: <code> var_value_tt: "There are [?num_of_dogs] dog(s) in the country"</code><br/>
By default, the scope where this is used gets assumed. Scoping works the same way as it does with variables regularly: <code> GER_var_value_tt: "There are [?GER.num_of_dogs] dog(s) in [GER.GetNameWithFlag]"</code><br/>
When the variable is a scope by itself, the question mark is still mandatory to use to distinguish from event targets: <code> target_tt: "The target country is [?target_country.GetName]"</code>. '''Variables can only be displayed in UI that supports dynamic localisation''', otherwise it will appear exactly as in the localisation file. List of such UI elements and necessary conditions for them to support dynamic localisation can be found in {{Section link|Localisation|Namespaces}}.

If the variable points to a certain address with more localised text, such as a country, namespaces can be used in the same manner as with a regular tag or event target: <code> most_dogs_tt: "[?most_dogs.GetNameDefCap] has the most dogs out of any country."</code> Question marks are still mandatory to include in this case. {{#section-h:Localisation|Formatting variables}}
By default, variable-related arguments such as <code>set_variable</code> or <code>check_variable</code> do not have a tooltip. For ''some'' of these arguments, such as <code>add_to_temp_variable</code>, it is possible to set a tooltip with an optional <code>tooltip = localisation_key</code> argument within. Others, such as <code>check_variable</code>, do not support this and a custom tooltip must be used instead, whether it's for [[Effect#Tooltip manipulation|effects]] or for [[Triggers#custom_trigger_tooltip|triggers]].

This tooltip argument is notable since the value of that localisation key allows using two [[Localisation#Nested strings|strictly-internal variables]] marked with dollar signs: <code>$LEFT$</code> and <code>$RIGHT$</code>.<br/>
<code>$LEFT$</code> represents the variable's value before the operation, while <code>$RIGHT$</code> represents the variable's value after the operation. This can be used to let the player know the current and/or the potential value of the variable in the same localisation key without taking up more space with <code>custom_effect_tooltip</code>.

Here's an example of it being done:
<pre>add_to_variable = {
    var = num_dogs
    value = ENG.num_cats
    tooltip = add_cats_to_dogs_tt
}</pre>This is the effect itself. Within localisation, this would be done:
<pre> add_cat_to_dogs_tt: "Adds [ENG.GetAdjective] cats to our $LEFT$ dog(s) to get $RIGHT$ in total."</pre>

=== Additional usage ===
By default, an unset variable is equal to zero when checked. However, the game also provides a [[wp:null coalescing operator|null coalescing operator]] for variables, used with a question mark. For example, <code>cost = var_name?100</code> within a decision will result in the cost of the decision being the same as var_name for the country. However, if that variable was never set or was cleared beforehand, then the decision will cost 100 {{icon|political power|1}} instead of 0.

Similarly to flags, a variable can be set as targeted by appending the target with <code>@TAG</code>. However, there are differences: in variables this does not append the target's ID to the name but stores it as a separate variable, as well as it being possible to target far more than countries, but also, for example, states or ideology groups. A variable reference <code>var_name_@GER</code> would resolve to a variable named <code>var_name_GER</code>. If called in a scope where PREV<code>= { tag = ENG }</code>, a variable reference <code>var_name_@PREV</code> would resolve to a variable named <code>var_name_ENG</code>. This can even be applied to temporary variables. You can also target variables using the syntax <code>var_name_@var:target_var_name</code>, regardless of the type of their value: be it country, state, token, even a raw integer. Variables targeting tokenizable types such as ideology groups and ideas provides a powerful alternative to scoped variables for associating a set of variable fields together, enabling one to get around the fact that ideas, ideologies, etc. do not have their own scopes. Generally, when you target a token, it is pasted in as its underlying integer ID (a number in the ten-thousands, usually), but this generally works fine, because it's consistent; the only problem is it's not human-readable. In general, variable targeting is substantially more optimized than meta-effect/trigger token pasting, not to mention more readable and understandable, so should be preferred wherever feasible.

It is technically possible, under some circumstances, to use multiple target specifiers in one variable name, usually strung together back-to-back at the end of the name, like <code>var_name_@THIS_@PREV</code>. However, this is extremely inconsistent, and the exact rules that govern how it behaves are currently an open question.

=== Usage examples ===
A common usage for variables is to grant a more 'complex' effect or a trigger. This is best done with temporary variables, as they can be set in either triggers or effects and they're not bound to scopes making operations on them more simple.<br/>
For example, the following can take 0.1% of every controlled state's population and add it directly to the manpower pool if the country has more democratic party support than fascist and communist party support combined:
<pre>if = {
    limit = {
        set_temp_variable = { fascom_pop = party_popularity@fascism }
        add_to_temp_variable = { fascom_pop = party_popularity@communism }
        check_variable = { party_popularity@democratic > fascom_pop }   # Completely hidden from the player.
        
    }                                                                   # 'limit = {}' is always hidden so it doesn't matter here, but use custom_trigger_tooltip when wanting the player to be aware.
    hidden_effect = {   # Every controlled state can look weird to the player otherwise.
        set_temp_variable = { temp_manpower = 0 }   # Optional, can be omitted as variables are assumed to be 0 if unset.
        every_controlled_state = {
            add_to_temp_variable = { temp_manpower = state_population_k } # Executed in the state's scope, so state_population_k is of the state. 
        }                                                                 # Temp variables are unscoped so it doesn't matter where they're edited.
    }                                                                     # For regular varibles, "add_to_variable" would need to scope into the correct country such as with "PREV.temp_manpower".
    add_manpower = var:temp_manpower # 'add_manpower = temp_manpower' also works
}</pre>
Since temporary variables can be set in triggers, they can be used as an alternative version of [[Triggers#count_triggers|count_triggers]], to count the amount of scopes where the trigger is fulfilled instead of counting the amount of fulfilled triggers:
<pre>if = {
    limit = {
        all_country = { # all_-typed scopes iterate over each entity until encountering a false, making them preferred over any_-typed ones that will exit out at a true.
            if = {                              # Direct limiting is only supported in effect scopes.
                limit = {
                    has_country_flag = example_flag
                }
                add_to_temp_variable = { flag_counter = 1 } # Assumed to be zero if unset
            }
        }
        check_variable = { flag_counter > 9 }
    }
    country_event = example.0 # The event fires if at least 10 countries have the example_flag flag.
}</pre>

Additionally, it is common for a variable to be a counter of something within a mechanic. For example, let CAN_beavers be a counter of beavers within the country.<br/>
When using variables in this sense, '''it is better to use [[Effect#Scripted effects|scripted effects]] to modify them''', in order to make changes to the variable easier. For example, a small gain in beavers could be the following:
<pre>CAN_beavers_small_add = {
    custom_effect_tooltip = CAN_beavers_small_add_tt    # Tooltip for the player to be aware.
    add_to_variable = { CAN_beavers = 100 } # Adds 100 beavers.
}</pre>
However, if 100 seems like too little, the scripted effect can be changed and it'll immediately apply the changes every single time that scripted effect was used without needing to find them manually. These effects may also share a single 'recalculating' scripted effect in common. This can be used to clamp the variable between a minimum and a maximum, to have an instant effect as soon as some variable value is reached, or to have another variable 'tied' to the value to be kept at an exact or a dynamic ratio. For example, these scripted effects can be used with the same CAN_beavers example:
<pre>CAN_beavers_calibrate = {
    clamp_variable = {  # The variable will be kept between these levels by calling this each time the variable is changed.
        var = CAN_beavers
        min = 0
        max = 2000
    }
    
    set_variable = { CAN_beavers_modifier = CAN_beavers }
    divide_variable = { CAN_beavers_modifier = 1000 }   # Another variable tied to the base value, as long as the scripted effect is called upon each change of CAN_beavers.
    
    if = {
        limit = {
            check_variable = { CAN_beavers = 0 }
        }
        country_event = CAN_beavers_extinction.0    # The event gets fired as soon as CAN_beavers hits 0.
    }
}

CAN_beavers_small_add = {   # Same as before, but with calling the previous effect
    custom_effect_tooltip = CAN_beavers_small_add_tt
    add_to_variable = { CAN_beavers = 100 }
    CAN_beavers_calibrate = yes
}

CAN_beavers_small_remove = { # Another example of a variable-modifying effect
    custom_effect_tooltip = CAN_beavers_small_remove_tt
    add_to_variable = { CAN_beavers = -100 }    # 'subtract_from_variable = { CAN_beavers = 100 }' also works.
    CAN_beavers_calibrate = yes
}</pre>
After creating these scripted effects, these can be used within any effect block to modify the variable at will. [[#Additional usage|Showing its value in localisation to the player]] would also be helpful, such as within a [[Decision modding|decision category]] or in [[Scripted GUI modding|scripted GUI]]. '''Variables cannot be used within idea modifiers directly, [[Modifiers#Dynamic_modifiers|dynamic modifiers can be used instead.]]'''

Remember that variables are not necessarily used as numbers only. For example, the simplest way to make the current ideology group to change to the one that's the most popular (aside from the ruling party) would be a <code>set_politics = { ruling_party = highest_party_ideology@exclude_ruling_party }</code>. Similarly, some event target-type variables can be used to cut down on code, such as <code>transfer_state = GER.capital</code> immediately transferring the capital of GER to the current scope.

=== MTTH variables ===
MTTH variables allow to dynamically create a [[AI modding#MTTH blocks|MTTH block]] to be used as a variable. These are stored within {{path|common/mtth/*.txt}} files, where each MTTH variable is a separate block, the name of which gets taken as the MTTH variable.<br/>
As per the name, the MTTH variables serve as a MTTH block, so everything that applies to MTTH blocks applies here: The starting base is defined with <code>factor</code> or <code>base</code> and then each <code>modifier = { ... }</code> begin a trigger block needed to be fulfilled for the <code>add</code> (for adding) or/and <code>factor</code> (for multiplying) within to be applied.
For example, the following is an example of a {{path|common/mtth/*.txt}} file:
<pre>my_mtth_variable_1 = {
    base = 10
    modifier = {
        factor = 10
        is_major = yes
    }
    modifier = {
        add = -10
        tag = FRA
    }
}
my_mtth_variable_2 = {
    base = other_variable
    factor = 0.5
    add = 10
}</pre>
This can be called the same way as a game variable, but somewhat differently: <code>set_variable = { my_var = mtth:my_mtth_variable_1 }</code>. This will check the MTTH block for the scope where it's checked and return the total result as a variable.

=== Token-valued variables ===
When one uses a literal of the form <code>token:some_tokenizable_value</code>, what is actually returned is an integer variable (commonly with a large value), with an internal flag set so that it is interpreted as an index into a central table of token strings that exists within the game's memory. This can be placed into a variable and passed around freely, it can be compared for equality to other token variables, and it can even be treated as a numeric in arithmetic even if usages for such cases are rare if nonexistent. Variables that have the token flag set can be distinguished in the output of the <code>list_vars</code> console command by dint of being automatically translated into their underlying raw-token strings. For example, <code>some_var_name = token:some_tokenizable_value (27012)</code>.

When the variable contains a token value, it is possible to use the <code>GetTokenKey</code> [[Localisation#Functions|localisation function]] (such as <code>[?some_var_with_token_value.GetTokenKey]</code>) to get the token string value of the variable, <code>token:</code> prefix not included. This is useful for pasting in [[Effect#Meta_effects|meta-effects]] and [[Conditions#Meta_triggers|meta-triggers]], however note that the <code>token:</code> prefix is still required if using it within the <code>set_(temp_)variable</code> operator. With some token-valued variables (specifically, those backed by named database objects of a tokenizable datatype), it is possible to use <code>GetTokenLocalizedKey</code> to retrieve the value of the localisation key with the same name as the token itself. This can be used to produce and/or define human-readable names, but it can also be taken advantage of to pass around arbitrary localisation values.

At present (1.12), known tokenizable datatypes include (but are probably not limited to):
* Ideas: These are generally the most flexible tokenizable datatype to use for a token that represents some arbitrary value by using a dummy idea. They can also carry an effectual payload in their <code>on_add</code> clauses, which can be used to pass around effects, or better yet, to return a set of variables encoding information relevant to the token value.
* Everything listed in {{path|common/script_enums.txt}}
* Ideologies/Ideology groups
* Technologies
* Equipment types - Sometimes this is a bit inconsistent with regards to the archetype/non-archetype distinction, so be wary.
* Operations
* Characters/Unit leaders: These are finicky in the game's core. When a character is promoted to a unit leader position, all tokenizations of that character's token will begin to return a different value, representing the token for the unit leader with the same identifier, which is linked to, but not the same as, the original character. The unit leader version of the token returns the same <code>GetTokenKey</code> and <code>GetTokenLocalizedKey</code>, but it is ''not numerically equal to the character version of the token.'' This can come into play if code compares the equality of two character-pointing variables.
* Decisions
* Buildings

=== Advanced use of tokens ===
Using tokenizable values in variables is the key to a number of advanced techniques that enable relatively seamless abstraction and metaprogramming, with a minimum of scripted-localisation boilerplate code. These techniques are, however, difficult to learn how to apply and can be unintuitive for those not already experienced. Therefore, caution is advised.

It is often useful to define related dummy tokenizable database entries, especially ideas, to represent arbitrary (possibly enumerable) custom datatypes. In most cases, ideas are preferable, however there can be uses for other token types, such as technologies if the code is to implement something that can't be done with a regular idea. The reason for idea preference is little to no downstream effects simply from existing, ease to define in bulk with minimal syntactic overhead, and ease to separate different custom datatypes by using custom idea categories and by placing the definitions for different custom datatypes in different files within {{path|common/ideas/}}. Decisions or technologies are common alternatives for custom datatypes with different usages, but generally datatypes other than these have too much downstream baggage (i.e. their very existence has consequences elsewhere, such as custom building types appearing automatically in certain UIs) to be worth using for dummy values. 

Using a systematic naming scheme, custom tokenizable datatypes can also convey multiple related localisation keys, similarly to the approach that the base game does for country naming with separate <code>TAG</code>, <code>TAG_DEF</code>, and <code>TAG_ADJ</code> keys. An example of that would be creating an idea <code>foo_bar</code> to act as the key identifier of the custom data value, whose localisation value carries the human-readable name of the data value, and another idea <code>foo_bar_description</code> ('''NOT <code>foo_bar_desc</code>, this collides with the localisation key generated for <code>foo_bar</code>''') to carry the localisation value for the description that would be associated with the data value identified by <code>foo_bar</code>. This description localisation can be accessed with a meta-effect or meta-trigger (including in scripted localisation), such as <code>set_temp_variable = { description_var = token:[IDEA_TOKEN]_description }</code> (where <code>IDEA_TOKEN = "[?idea_var_name.GetTokenKey]"</code>). This allows accessing <code>[?description_var.GetTokenLocalizedKey]</code> in localisation where <code>description_var</code> is visible, such as the <code>localization_key</code> following the scripted localisation <code>trigger</code> block where the pasting meta-trigger is called.

As far more advanced usage, it is possible to rig up the dummy tokenizable values with arbitrary localisation payloads that carry fragments of Clausewitz code which can then be pasted together in meta-effects (or triggers) to approximate true token string manipulation or at least concatenation, as there's presently no way to split strings, token or otherwise. This is probably most useful for encoding a set of suffixes that can be applied to an idea token to dynamically navigate within the custom datatype's naming scheme. For example, if the mod has a custom datatype representing positions within a government such as the Secretary of State and the Attorney General, such that it might have a base idea <code>position_attorney_general</code>, a corresponding description idea (suffixed with <code>_description</code>), a singular name idea (suffixed with <code>_singular</code>, carrying the name of the data value in singular form, like "Attorney General"), and a plural name idea (suffixed with <code>_plural</code>, carrying the name in plural form, like "Attorneys General"), it is possible to select which name to use by having a dummy idea called <code>enum_position_name_singular</code> with the localisation value <code>"_singular"</code>, and another called <code>enum_position_name_plural</code> with the localisation value <code>"_plural"</code>. After that, the mod may pass one of the <code>enum_position</code> ideas in a variable and then paste its <code>GetTokenLocalizedKey</code> after the <code>GetTokenKey</code> of the base idea in a meta-effect/trigger to produce either <code>token:position_attorney_general_singular</code> or <code>token:position_attorney_general_plural</code>. This is a contrived example, and is also possible to be done using a scripted localisation system with hard-coded entries to access the singular vs. plural ideas, but this approach has far more use in more complex systems where creating a scripted localisation file can prove to be far too bulky.

From 1.12 onwards, anything defined in {{path|common/script_enums.txt}} can be freely tokenized, even if defined in a totally custom enumeration block, seperate from the ones hard-coded into the game. It is therefore theoretically possible to define arbitrary tokens of a custom enum type in this file. However, for compatibility reasons, using <code>script_enums</code> for custom enumeration types should be avoided, as this will ensure that the mod will not be compatible with any other mod that overwrites that file, no matter which mod gets priority on the file. As before, it's preferable to have a custom idea category with a set of dummy ideas in their own file that will not be overridden by other mods representing the values the custom enum can take on. This pattern is a useful replacement for passing around arbitrary integers that have some enumerable meaning when interpreted by code, especially if those integers are interpreted to yield names in scripted localisation, in which case this pattern can substantially remove the need for scripted localisation boilerplate by allowing defining the localised name for each enum member directly in a localisation file, as similarly to defining the localised name for an idea.

Technically, any raw token used in the language can actually be tokenized: for instance, <code>token:set_stability</code> (that is, a token literal of the name of the [[Effect#Mana|set_stability effect]]) is perfectly valid, and yields a token for which <code>GetTokenKey</code> and <code>GetTokenLocalizedKey</code> both return <code>set_stability</code>. However, this is rarely useful (especially since it does not apply to scripted effect and trigger names), and arguably counts as a bug; it mostly just provides an interesting peek into the internals of the tokenization system, indicating that it most likely delegates to a pre-existing string de-duplication system built into the Clausewitz engine.

== Arrays ==
Arrays are similar to variables, but they instead store a collection of variables.<br/>
An array in the game is structured with indexes starting from 0. <code>arrayname^0</code> as a variable refers to the first item of <code>arrayname</code>, <code>arrayname^1</code> refers to the second item in the array, and so on. In order to obtain the total amount of items within the given array, <code>arrayname^num</code> is used. An array is assumed to have no elements by default, if not already created, with [[#add_to_array|add_to_array or add_to_temp_array]] usually being used to create an array.

Temporary and regular arrays exist, with the same distinction between them as in variables: temp arrays can be set and modified in triggers and effects, are unscoped, and cleared after the effect/trigger block ends; regular arrays can only be set or modified in effects, are linked to a specific scope, and persist forever until cleared manually.

A single element of an array is considered a variable in every regard: <code>set_variable = { arrayname^0 = 10 }</code> will set the first element to be equal to 10 (with <code>set_temp_variable</code> used for temporary arrays), <code>check_variable = { arrayname^0 > 10 }</code> is also valid syntax, and <code>[?arrayname^0]</code> will print the value of the element in localisation. However, the element must already exist for it to be edited. '''There is no simple way to print an entire array in localisation''', however a recursive scripted localisation can be used, which modifies the value of a temporary variable by one until hitting the array's size each time that it's used.

|Runs the effects as long as the limit is fulfilled.
|The limit is only checked at the beginning and once the effect block is executed. Can't run for more than 1000 times by default<ref name="loop"/>. Comparison types are the same as used in [[#check_variable|check_variable]].

=== Scorers ===
Scorers are defined within {{path|common/scorers/country/*.txt}} files, serving as a sorted list of targeted countries, based on how well they did in the specified [[AI modding#MTTH block|MTTH block]]. The scorer has the primary country which the scorer is used for and targeted countries.

Each scorer is a separate block within the file, with the name of the block being the name of the scorer. The scorer then has the <code>targets = { ... }</code> block that encompasses everything else about the scorer.

Scorers target countries similarly to [[Decision modding#Targeted decisions|targeted decisions]] but with differences, with the following arguments being used within of <code>targets = { ... }</code>:

As with targetted decisions, [[Scopes|FROM]] is the scope of the country being targeted by the scorer, while ROOT is the scope of the country invoking the scorer.

Scorers do not seem to hotreload very reliably. But, assuming you're changing an existing file rather than making a brand new one, edits to a scorer are reflected if you quit to main menu and then re-enter the game.
* <code>targets = { TAG, TAG }</code> - A list of country tags, including other [[Scopes#Dual scopes|dual scopes]] such as [[Scopes#overlord|overlord]], that can be targeted by this scorer. Unlike targeted decisions, this seems to be comma-separated.
* <code>target_array = global.majors</code> - The array to target. The default scope is the primary country.
* <code>target_root_trigger = { ... }</code> - The trigger block which the primary country must fulfill for the target to work. In here, ROOT (default scope) is the country running the scorer.
**It's not clear if there's any difference between <code>target_root_trigger</code> and <code>target_trigger</code> except that <code>target_root_trigger</code> is presumably only run once when invoking a scorer, and FROM cannot be used within it.
* <code>target_trigger = { ... }</code> - The trigger block that target country must fulfill for the target to work. In here, ROOT (default scope) is the country running the scorer and FROM is the potential target of the scorer.
* <code>targets_dynamic = yes</code> - If set, makes the dynamic countries possible targets. Defaults to no.
* <code>target_non_existing = yes</code> - If set, makes non-existing countries possible targets. Defaults to no.
* <code>score = { ... }</code> - This is the MTTH block itself that ranks the countries in the list. By default, THIS (the default scope) is the target, FROM is the primary country initialising the check.
For example, the following is a scorer that sorts every major country depending on the amount of states they own:
<pre>asian_countries_owned_states_scorer = {
	majors_owned_states = {
		targets = {
			target_array = global.countries
			target_trigger = {
				FROM = {
					capital_scope = {
						is_on_continent = asia
					}
				}
			}
			score = {
				modifier = {
					add = num_owned_states
				}
			}
		}
	}
}</pre>
After defining a scorer, there is the way to use it: an argument, similar to arrays and variables. Similarly to them, regular and temporary versions exist, which decides whether the variable/array it's writing to is regular or temporary. Both can only be used inside of country scope. Regular versions can only be used in effects, and temporary versions can be used in effects and triggers.

==== Example ====
Scorers can be helpful to create a sorted list of scopes by a certain element, such as political power. For example, the following can be used to find the countries with the most infantry equipment.

Within the scorer file:
<pre>infantry_equipment_scorer = {
	targets = {
		target_array = global.countries
		targets_dynamic = yes
		score = {
		    modifier = {
		    	add = num_equipment@infantry_equipment
		    }
		}
	}
}</pre>
Within an effect block:
<pre>ROOT = {
	get_sorted_scored_countries_temp = {
		scorer = infantry_equipment_scorer
		array = scored_array
		scores = scored_values
	}

	for_loop_effect = {
		end = scored_array^num
		value = s
		log = "[?scored_array^s.GetName] had inf equipment [?scored_values^s]"
	}
}</pre>

== Game variables ==
{{SVersion|1.13}}
Game variables are read-only variables that are used to read certain aspects of the game, such as the current political power that the player has. These cannot be modified using any variable-modifying argument, but they can serve as the value within them (such as <code>add_to_temp_variable = { temp = GER.army_experience }</code> adding the amount of army experience that GER has to the <code>temp</code> temporary variable). Some are targeted, requiring one more argument in the target's place. For example, <code>modifier</code> has a modifier token as the target, so it would be used as <code>set_temp_variable = { temp = modifier@consumer_goods_factor }</code>, where <code>consumer_goods_factor</code> is the name of the modifier used as the argument.

Remember that to get global-scoped game variables, it is necessary to scope into global as <code>global.year</code> rather than leaving out the scope. A list of game variables can be found within {{path|documentation/dynamic_variables_documentation}}. A large portion of purely numeric [[Triggers]] also have corresponding game variables that are not mentioned in the documentation. The following game variables exist in the game (Some duplicates are omitted):
