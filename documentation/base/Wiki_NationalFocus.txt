{{Version|1.13}}

National focus trees are defined within {{path|common/national_focus/*.txt}} files. Like most other files, the filename is irrelevant and isn't used for anything other than organisation. One file can store more than one focus tree or none at all.
== Focus tree ==
{{SVersion|1.13}}
A focus tree is defined by using a <code>focus_tree = { ... }</code> block. The following arguments are used:

<code>id = my_focus_tree</code> decides the ID that the focus tree uses. It is mandatory to define, and an overlap will result in an error. The ID is primarily used for the [[Conditions#has_focus_tree|has_focus_tree trigger]] and the [[Effect#load_focus_tree|load_focus_tree effect]], whose tooltips use it as a localization key.

<code>country = { ... }</code> is a [[AI modding#AI will do|MTTH block]] that assigns a score for the focus tree, deciding which one is used in-game. This is evaluated ''before the game's start'' and the check is essentially '''never refreshed'''{{cref|a}}. The focus tree with the highest score will be the one that gets loaded for the country. By default, the score starts with 1. A typical usage looks like this:
<pre>country = {
    factor = 0
    modifier = {
        add = 20
        original_tag = TRA
    }
}</pre> In this case, countries originating from {{flag|Transylvania}} (i.e. the country itself and any civil war or collaboration government breakaways, as ensured by [[Conditions#original_tag|the original_tag trigger]]) will have the score of 20, while every other country will have the score of 0. Assuming that there is no other focus tree where {{flag|Transylvania}} has a higher country score, this will ensure that this focus tree gets loaded for it.

<code>default = yes</code> sets the focus tree to be marked as default. In total, ''there should be one total default focus tree'', no more, no less. A focus tree being marked as default means that if every other focus tree has a country score of 0, this tree will be chosen instead. Additionally, a country starting with a focus tree will fail to appear in the "minor countries" section within the "interesting countries" menu before the game's start. If this is left out from a focus tree, it gets assumed to be non-default.

<code>reset_on_civilwar = no</code> is not determined on its effect. Instead, this is how focus trees are handled in civil wars, regardless of if <code>reset_on_civilwar</code> is set or how it's set:<br/>
When a civil war starts, the original country will always continue using the focus tree. The focus it's doing will not be paused or cancelled by the civil war itself. The revolter will have the focus tree it's using evaluated when the civil war starts, assigning one depending on each tree's <code>country = { ... }</code> value. If the same focus tree gets used for the revolting country as the one that the original country used when the civil war started, every focus that the original country has completed will get completed for the revolting country, including setting the same focus progress for the one that's being completed by the original country at the moment. Otherwise, the focus progress will get lost.

<code>shared_focus = TAG_focusname</code> will set the focus tree to include the specified [[#Shared focuses|shared focus]] and every focus that is connected to it via prerequisites. '''Setting this to a non-existing focus causes a game crash when loading into the main menu.'''

<code>continuous_focus_position = { x = 1200 y = 100 }</code> is the position of the top left corner of the continuous focus menu ''in pixels''. For comparison, by default{{cref|b}}, the continuous focus palette has the position of 50 on the X axis and 1000 on the Y axis. If both x and y are set to 0 or the position is undefined for the tree, it resets to the default position.

<code>initial_show_position = { ... }</code> decides the initial position of the camera when the focus tree is first opened. There are 2 ways to arrange it:<br/>
* <code>focus = TAG_focusname</code> will make the camera centre on the specified focus in particular. It'll be in the top centre of the screen exactly, taking offsets into consideration.
* <code>x = 12 y = 0</code> decides the exact position of the top-centre of the camera. This uses the same coordinate system as regular focuses do, by default a unit of x being equal to 96 pixels and a unit of y being equal to 130 pixels{{cref|c}}<br/>
: This also accepts <code>offset = { ... }</code>, adding the specified values to respective positions if the conditions within the <code>trigger = { ... }</code> trigger block are met for the country. For example, this will apply the modifier and result in a position of x = 13, y = 1 if the country is BHR:
<pre>initial_show_position = {
    x = 17
    y = 0
    offset = {
        x = -4
        y = 1
        trigger = {
            tag = BHR
        }
    }
}</pre>

<code>focus = { ... }</code> are the focuses themselves. Each focus that's put within the focus tree will. The focuses '''have''' to be within a <code>focus_tree = { ... }</code> in order to let the game know which focus tree exactly to assign them to. If a <code>focus = { ... }</code> blocks ends up outside of a <code>focus_tree = { ... }</code> or within another <code>focus = { ... }</code>, this gets marked within the [[Modding#Advantages to using debug|error log]] as "focus" being an unexpected token, fixed by adjusting brackets as needed.

=== Shortcuts ===
Shortcuts are used to add navigation buttons in the lower left corner that, when clicked, will move the camera to a specific focus.

Parameters:
* <code>name = lockey</code>, the localized name that will be displayed on the button.
* <code>target = focus_id</code>, focus where the camera should be moved.
* <code>scroll_wheel_factor = <float></code>, zoom, the higher the value the greater the zoom. (between 0 and 1)
* <code>trigger = { ... }</code>, a trigger block specifying whether the shortcut button should be visible, optional. (If true, it will not supersede allow conditions of target focus.)
An example shortcut definition is as such:
<pre>shortcut = {
  name = GER_oppose_hitler_shortcut
  target = GER_oppose_hitler_ww 
  scroll_wheel_factor = 0.485
  trigger = {
    has_dlc = "Gotterdammerung"
  }
}</pre>

=== Inlay window ===
Focus inlay windows are generic inlay components that can be added to the focus tree, similar to the continuous focus window, a [[Scripted_GUI_modding|scripted GUI]] object. Currently, only support showing information and no interaction.

==== Definition ====
Definitions for inlay windows are located in {{Path|common/focus_inlay_windows/*.txt}}.

For a guide written by the game developers, see [https://forum.paradoxplaza.com/forum/developer-diary/developer-diary-performance-modding.1713814/ Dev dairy modding]

Parameters:
* <code>window_name = gui_component_name</code> the name of the scripting GUI to use.
* <code>internal = <bool></code> if true, then the inlay window is only visible to the country itself. (defaults no)
* <code>visible = { ... }</code> whether the inlay window should be visible, when not visible, no evaluations will be done.
* <code>scripted_images = { ... }</code> list of images that should have dynamic sprites.
** <code><icon_name> = { ... }</code> name of the icon (must be a subcomponent of "gui_component_name"). List of possible gfx:es for the icon, first that evaluates to true is selected.
*** <code><gfx_name> = { ... }/yes</code> If a trigger is provided, then it will be evaluated with the country scope of the focus tree. If "yes" is set, then it will always be used. Note: "yes" is commonly the last entry in the list that acts as a default case.

An example for inlay window definition:
<pre>inlay_window_id = {
  window_name = gui_component_name
  internal = yes
  visible = {
    visibility_triggers
  }
  scripted_images = {
    advisor_portrait = {
      gfx_name = {
        has_completed_focus = TAG_focus_name_example
      }
      gfx_name_fallback = yes
    }
  }
}</pre>

==== Declaration ====
Inlay window must be declared in focus tree. Any number of inlay windows can be defined for focus tree.

Parameters:
* <code>id = inlay_window_id</code> inlay window identifier.
* <code>position = { x = <int> y = <int> }</code> position on the tree, same syntax and scale as continuous focus position.
An example declaration of inlay window is as such:
<pre>inlay_window = {
  id = ger_inner_circle_inlay_window
  position = { x = 4500 y = 1150 }
}</pre>

== National focuses ==
{{SVersion|1.13}}
The ID for the focus is defined using <code>id = TAG_focusname</code>. While it's optional to preface the focus' ID with the country's tag, doing so is preferred to avoid overlapping focus IDs from different focus trees. Having the same focus ID in different focus trees leads to errors, such as broken prerequisite lines and effects or triggers such as [[Effect#complete_national_focus|complete_national_focus]] or [[Conditions#has_completed_focus|has_completed_focus]] not working correctly. Instead, it could be possible to use [[#Shared focuses|shared focuses]] to put the same focus in different focus trees.
=== Name and description ===
The name of the focus depending on the language that's turned on is defined within {{path|localisation/}}, using the ID of the focus as the localisation key. For English in particular, this is defined within any {{path|localisation/english/*_l_english.yml}} file with the UTF-8-BOM encoding. It is preferable to use new localisation files when possible rather than overwriting base game localisation in order to not have to change that for compatibility with recent versions, and to do so, the file should have a new name that doesn't exist in base game, but it must still end with <code>_l_english.yml</code> to be loaded properly. The focus' ID is used as the localisation key used to establish the name in the enabled language, while a description uses the focus' ID with <code>_desc</code> appended:
<syntaxhighlight lang = "yaml">l_english:
 TAG_focus_name: "Example focus"
 TAG_focus_name_desc: "Example focus description"
</syntaxhighlight>

=== Position ===
The position of the focus is decided via <code>x = 5</code> and <code>y = 1</code> attributes. By default, a unit of x is equal to 96 pixels and a unit of y is equal to 130 pixels{{cref|c}}. In other words, a focus directly below another focus would have a unit difference of 1, while a focus directly to the right of another one would have a unit difference of 2. By default, this is relative to the top left corner of the tree: a larger x value moves the focus right, a larger y value moves the focus ''down''.

It is also possible '''and preferred''' (on focuses with prerequisites) to make the focus' position be relative to another focus with doing <code>relative_position_id = TAG_other_focus</code>. This will position the focus relative to that focus, adding the <code>x</code> and <code>y</code> values to the other focus' position (after calculating that one's relative_position_id too). Doing so allows for more flexibility in the focus tree design by allowing to easily modify the position of the entire branch at the same time, due to updates to the children focuses' positioning. This also allows to only use the later <code>offset = { ... }</code> only in the top focus of each branch that requires to be moved.<br/>
For example, if focus A has <code>x = 1  y = 2</code>, focus B is positioned relative to focus A and has <code>x = 3  y = 4</code>, then the focus B will be positioned, in total, 4 steps to the right and 6 steps down of the top-left corner. A focus C positioned relative to focus B at <code>x = -1 y = 1</code> would then be located 3 steps to the right and 7 steps down of the top-left corner.<br/>
The game can behave unstably with an incorrect relative position ID. A recursion (such as a focus being positioned relative to itself or focus A and focus B being positioned relative to each other) may cause a game crash since it is impossible to determine the exact position of the focus, and the focus must also be located in the same focus tree for the argument to work properly.


Changing a focus' position based on a condition being met is done with <code>offset = { ... }</code>. The <code>x = 10</code> and <code>y = -3</code> values will be added to the focus' position if the conditions within <code>trigger = { ... }</code> are met for the country ''when the focus tree is loaded''. This looks like the following:
<pre>offset = {
    x = -5
    trigger = {
        has_dlc = "Poland: United and Ready"
    }
}</pre>This in particular will move the focus 5 units to the left if the "Poland: United and Ready" DLC is turned on. This check also can be refreshed mid-session with the [[Effect#mark_focus_tree_layout_dirty|mark_focus_tree_layout_dirty effect]]{{cref|d}}, applying the offset if true.

=== Interaction with other focuses ===
{{anchor|prerequisite}}
<code>prerequisite = { focus = TAG_other_focus }</code> decides the focuses necessary to complete for this focus to be available. At least one focus within a prerequisite has to be completed to mark the prerequisite as true, and each prerequisite much be completed to take the focus. If neither of the prerequisites is located in the same focus tree, then the focus will not appear. In other words, an OR statement is done by putting 2 focuses inside a prerequisite as <code>prerequisite = { focus = TAG_other_focus_1 focus = TAG_other_focus_2 }</code>, while an AND statement is done by putting two different prerequisites like the following:
<pre>prerequisite = { focus = TAG_other_focus_1 }
prerequisite = { focus = TAG_other_focus_2 }</pre>
This system cannot represent every boolean logical arrangement, such as <math>A \lor (B \land C)</math> (Where <math>A</math>, <math>B</math>, and <math>C</math> represent whether a focus is complete) or with anything using negation. In this case, it can be possible to, instead, put an OR statement for either of the focuses necessary to complete this one and use the [[Conditions#has_completed_focus|has_completed_focus]] trigger within the <code>available = { ... }</code> block with necessary flow control tools. [[Conditions#custom_trigger_tooltip|A custom trigger tooltip can be used to make it easier for the player to understand.]]

{{anchor|mutually_exclusive}}
<code>mutually_exclusive = { focus = TAG_other_focus }</code> makes this focus impossible to select if the specified focus has been completed. If both focuses are mutually exclusive toward each other, then the mutually exclusive arrows will be shown in the focus tree view. Mutual exclusivity to multiple focuses is usually done by putting several of <code>focus = TAG_focusname</code> in the same mutually_exclusive, but defining several of mutually_exclusive is also possible.

Neither prerequisites nor mutual exclusivity require the other focus to be in the same focus tree. This means that it can be used with [[#Shared focuses|shared focuses]] to declare a regular, non-shared focus as mutually exclusive or a prerequisite without any errors, even when used in a focus tree not containing that focus.

The difference between prerequisites and using [[Conditions#has_completed_focus|has_completed_focus]] within the <code>available = { ... }</code> block is that the prerequisites show up as lines within the national focus tree view and show up separately from other triggers in the tooltip of a focus.

There is an issue that leads to prerequisite lines not working properly: duplicate focus IDs within different focus trees. In this case, the game can take the position of the focuses as the ones within the different focus tree that contains the same focuses, leading to them appearing to link towards empty spaces or start inside of them. This can also break the path-generating algorithm and make it use the wrong turn sprites, which will show up broken even in trees that don't have any duplicates.<br/>
In order to avoid this, duplicate focus IDs must be avoided. A simple way to decrease the chance drastically is to preface the focus IDs with the country tag (such as <code>TAG_focus_name</code>) or something else that's unique for the focus tree (Such as <code>REGION_focus_name</code> for a shared regional tree). If the same focus tree should be used for several countries, this can be done by only having one focus tree where the <code>country = { ... }</code> of the tree is set up so that the desire to use it is the highest for several countries instead of just one; if the same focus tree branch should be used within several different focus trees, then [[#Shared focuses|shared focuses]] can achieve exactly that.

Additionally, the game intends for a focus' prerequisite to be placed above the focus that requires it and it is unable to correctly generate the path to the focus otherwise, which will show up as having a path that uses the wrong sprites in the 90° turns.

=== Icon ===
{{Sprite overview}}
Within a focus, an icon is assigned with the line of <code>icon = GFX_focus_icon</code>. This assigns two sprites to the focus, in particular:
* Regular sprite, with the name of <code>GFX_focus_icon</code>. This is used in the focus description view and in the focus tree view when the focus is unavailable, is being completed, or has been completed.
* Sprite for the shine animation, with the name of <code>GFX_focus_icon_shine</code> (With _shine at the end). This is used in the focus tree view for focuses that are currently available and in the country politics and diplomacy views for the focus currently being complete. Since the game [[Modding#Loading_files|doesn't use filenames in evaluation]], putting the spriteType definition into a file with <code>shine</code> in the filename isn't either necessary or sufficient. Only the name of the sprite is used, which must be the same as the regular sprite with _shine appended to the end. '''The most common mistake when the shine doesn't work is not following this naming rule'''.
If one of these is undefined or is defined incorrectly, the [[File:Goal_unknown.png|28px]]missing focus icon will be used instead of the appropriate sprite, however the working sprite will continue to be used.<br/>
If the texturefile links to a non-existing file, whether it's the folder path that's incorrect{{cref|e}} or the filename, including the extension, the focus icon will appear as fully transparent.<br/>
By default, the base game stores images for focus icons in the {{path|gfx/interface/goals/}} folder and sprites in the {{path|custom=1|interface/goals.gfx}} and {{path|custom=1|interface/goals_shine.gfx}} files. Since there's no reason to copy the files to the mod and doing so will lead to needing to update the file after a major update to use new sprites, it's best to create a new file in the folder for sprite definitions.<br/>

=== Dynamic appearance ===
<code>dynamic = yes</code> allows the title and icon of the focus to be dynamic. In particular, it allows using [[scripted localisation]] in focus titles and a dynamic icon. If false or unset, it will still allow using them, but instead it will pick one at the game's start and never change it.

Dynamic icons are treated similarly to events' titles or descriptions: instead of a single <code>icon = GFX_focus_example</code> a focus can contain multiple <code>icon = { ... }</code> at the same time, each with a unique trigger. The first possible icon will get used in the focus. An <code>icon = { ... }</code> consists of a <code>trigger = { ... }</code>, which must be fulfilled by the country for the icon to be selectable, and <code>value = GFX_focus_example</code>, which determines the 2 sprites used for the focus.

An example dynamic focus definition, with all other optional arguments omitted:
<pre>focus = {
    id = TAG_political_focus # In localisation, TAG_political_focus: "[THIS.GetRulingIdeology] political focus"
    icon = {
        trigger = { has_government = democratic }
        value = GFX_focus_example_democratic
    }
    icon = {
        value = GFX_focus_example_fallback
    }
    dynamic = yes
}</pre>
=== Triggers ===
{{See also|Triggers}}
In order to take the focus, aside from the focus prerequisites, the conditions within the <code>available = { ... }</code> block must be met. This functions as an [[Scopes#AND|AND block]], so each of the triggers must be true to fulfilled. [[Scopes]] can be used to check for conditions for other countries or within states. By default, the scope is of the country doing the focus. For example, this example requires the country to have more than 10% {{icon|Stability|1}} and for the state 294 to be owned by the {{flag|Republic of Qatar}}:
<pre>available = {
    stability > 0.1
    294 = { is_owned_by = QAT }
}</pre>

<code>bypass = { ... }</code> is similar, but for bypassing the focus. Bypassing a focus marks the focus as complete, but does not grant its effects within the completion reward, instead focus can have <code>bypass_effect = { ... }</code>, focus can have <code>enable_automatic_bypass = no</code> to prevent bypassing focus automaticly. The exact same applies as to <code>available</code>: it's an AND block that assumes the country doing the focus by default. <code>bypass_if_unavailable = yes</code> can be used to make the focus automatically bypass as soon as the <code>available = { ... }</code> block is not met without needing to port over the triggers.

{{anchor|allow_branch}}<code>allow_branch = { ... }</code> is used to tell when the focus should be visible. '''This is only checked when the focus tree is first loaded'''. However, this check also can be refreshed mid-session with the [[Effect#mark_focus_tree_layout_dirty|mark_focus_tree_layout_dirty effect]]{{cref|d}}, making the focus visible or invisible depending on if it's true or not. By default, a focus will also be disallowed if either of the parent focuses (as set by prerequisites) is disallowed. However, a focus containing <code>allow_branch</code> within its definition will only check its own allowed status, still showing up if it has disallowed parents. This means that if a branch is set to be disallowed under certain conditions, the first focus in any sub-branch that has its own conditions for being allowed must also contain the parent branch's condition within of itself, while other focuses may not have any <code>allow_branch</code> defined at all due to it being inherited from parents.

<code>available_if_capitulated = yes</code> sets the focus to be possible to complete while being capitulated. By default, this is set to false.

<code>cancel_if_invalid = no</code> and <code>continue_if_invalid = yes</code> decide how to treat the focus if the <code>available = { ... }</code> block becomes false while doing it. By default, these are true and false respectively. If both are set to false, the focus would pause when the <code>available = { ... }</code> block is false. This will not remove the gain_focus [[Modifiers#static modifiers|static modifier]], which by default results in costing 1 {{icon|political power|1}} per day when doing the focus.

<code>cancel = { ... }</code> decides additional conditions which would cancel the focus if met. This is usually paired with <code>cancel_if_invalid = no</code>.

<code>historical_ai = { ... }</code> decides when the AI is able to pick this focus with the historical focus turned on. This does not ensure it picks this focus, rather prevents it from picking it when false. This takes priority over the order of focuses granted within [[#AI strategy plans|AI strategy plans]]: if the AI were to do this focus next by the plan, yet <code>historical_ai = { ... }</code> is false and historical focus is turned on, then it won't be able to.

=== Effects ===
{{See also|Effects}}

The primary reward of the focus is done with <code>completion_reward = { ... }</code>. This executes each [[effect]] within the specified [[scopes]] in order that they are put in the file. The assumed scope is the country doing the focus. For example, this would add 100 {{icon|political power|1}} to the country doing the focus and fire the <code>my_event.0</code> country event to {{flag|Oman}}:
<pre>completion_reward = {
    add_political_power = 100
    OMA = { country_event = my_event.0 }
}</pre>

The tooltip of the focus can be changed with <code>complete_tooltip = { ... }</code>, ''which is also an effect block''. This would be equivalent to putting the contents of the reward inside of [[Scopes#hidden_effect|hidden_effect]] and using [[Scopes#effect_tooltip|effect_tooltip]] in the same reward. This can be useful if the tooltip of the reward appears cluttered. For instance, using [[Scopes#random_owned_controlled_state|random_owned_controlled_state]] thrice with the same effect in each one can result in a cluttered tooltip, as each state and its effects would appear individually. But if, instead, the code [[Effect#set_state_flag|sets a state flag]] for each state in the reward and, in the tooltip, uses [[Scopes#every_state|every_state]] limited to the ones that have the state flag, it'll show the same effect being executed for 3 states at the same time, cutting it into a third of what it was.

Additionally, <code>select_effect = { ... }</code> is used to execute an effect when the focus is ''selected''. This has no tooltip shown to the player. This also automatically makes the focus impossible to cancel manually. It may still be cancelled automatically if set to do so, so in most cases it'd be preferable to set it to not cancel if invalid in order to prevent the effects within from firing more than once, as there is no way to execute an effect when the focus gets cancelled.

Additionally, <code>bypass_effect = { ... }</code> is used to execute an effect when focus is [[#Triggers|''bypassed'']].

'''The focus is not yet marked as complete when the effects are being executed.''' What this means is that any [[Conditions#has_completed_focus|has_completed_focus check]] ran within will return as false. This can be a hinderance in some cases, most commonly when using allow_branch (as the effect to refresh the check will not work properly when put within a focus). Some alternatives can be considered:
* [[Data structures#Flags|Country flags to track that a focus has been completed]]. Since country flags are set immediately, as long as the setting precedes the requirement, this will work.
* Delaying the effects that would require the focus to be complete. This is usually done by firing a hidden event. If done with no delay, the event will be fired ''right after'' the focus is complete, resulting in no noticeable delay for the player, but still executing the effects in order.
* Forcefully marking the focus to be complete. This, for example, can be done by using [[Effect#load_focus_tree|load_focus_tree]] for the same focus tree with the completed focuses being kept. Doing so will not interrupt the completion reward's execution.

=== Search filters ===
{{SVersion|1.13}}
Search filters for each focus are set with <code>search_filters = { ... }</code>. Within that block, each search filter that the focus has is put inside, separated by whitespace characters. For instance, the following will assign FOCUS_FILTER_MANPOWER and FOCUS_FILTER_POLITICAL to the focus: <code>search_filters = { FOCUS_FILTER_MANPOWER FOCUS_FILTER_POLITICAL }</code>. These filters get used in the search menu in the top right of the national focus tree view.

A focus filter is not defined in any file, but instead they are created dynamically for each focus tree. Each one uses the sprite the same as its name but with <code>GFX_</code> inserted in the beginning. For instance, this defininition within any {{path|interface/*.gfx}} file would be used for FOCUS_FILTER_MY_MOD:
<pre>spriteType = {
    name = GFX_FOCUS_FILTER_MY_MOD
    texturefile = gfx/interface/focusview/filter/my_mod_icon.dds
}</pre>As before with regular focus icon, the exact folder where filter icons are stored is irrelevant, as long as the texturefile specified within the sprite is correct.<br/>

=== Titlebar styles ===
Styles are used for determining the picture used for the titlebar, the background on which the focus name is shown.
A titlebar is set by using <code>text_icon = mod_focus_style</code> in the focus.

A style that can be used in focuses is created by using a root-level <code>style = { ... }</code> block in a {{path|common/national_focus/*.txt}} file, by default <code>00_titlebar_styles.txt</code>. There are these possible attributes that can be used inside of a style:
* <code>name = example_style</code> is the name of the style. The argument determines what needs to be written in a focus' <code>text_icon</code> to reference this style.
* <code>default = yes</code>, if written, will make any focus without a defined <code>text_icon</code> use this style. Only one style total may use this.
* <code>unavailable = GFX_focus_unavailable_example</code> is the sprite that gets used when the focus is unavailable.
* <code>completed = GFX_focus_unavailable_example</code> is the sprite that gets used when the focus has been completed.
* <code>available = GFX_focus_unavailable_example</code> is the sprite that gets used when the focus is available for completion, but not yet selected.
* <code>current = GFX_focus_unavailable_example</code> is the sprite that gets used when the focus is currently in progress of being completed.

=== AI will do ===
<code>ai_will_do = { ... }</code> is a [[AI modding#AI will do|MTTH block]] that decides the likelihood for the AI to do this focus if [[#AI strategy plans|an AI strategy plan]] is not set.<br/>
By default, each focus has a score of 1. The arguments of <code>base</code> (changing the value), <code>add</code>, and <code>factor</code> (multiplying it) can be used to modify it.<br/>
Within the <code>ai_will_do = { ... }</code> block, <code>modifier = { ... }</code> functions as a trigger block where the prior three value-modifying arguments are also supported. The value will be modified if the triggers are true. For example, the following will result in the value of 15 for POL and a value of 5 for every other country:
<pre>ai_will_do = {
    base = 5
    modifier = {
        factor = 3
        tag = POL
    }
}</pre>
An arbitrarily large amount of modifiers is possible to add to an ai_will_do, and they will apply in the order they're put in the code. It is also possible to use [[variables]] within a modifier of the ai_will_do value.

The way that the value is evaluated for AI picking the focus is that, when picking a focus to do, it generates a random decimal value between 0 and the ai_will_do value for each of the focuses. If the evaluated focus has a prerequisite focus that the AI has just completed, the generated ai_will_do value gets multiplied by {{Read define|NAI|FOCUS_TREE_CONTINUE_FACTOR|ref=yes|beforeref=before the game picks a focus.|spacebeforeref=yes}} If the [[#AI strategy plans|AI strategy plan]] that the country is currently following has <code>focus_factors = { ... }</code> defined for this focus, the value gets multiplied by the specified value. Afterwards, the game picks the focus that has the highest generated value. If neither of the focuses has a value above 0, the AI will not pick any of them, instead going into continuous focuses if possible or not doing any otherwise.<br/>
'''Due to that algorithm, low values are less likely to be picked than intuition suggests.''' 

To reiterate, '''this is only evaluated if the [[#AI_strategy_plans|AI strategy plan]] for the country doesn't have the order of the focuses set''' or if none of the focuses in that order can be followed.Comparing the chances between focuses is the following:<br/>
==== Formulas for chance calculation ====
As the general formula can be complex to calculate without a special tool, simpler calculations for special cases can provide quite useful: whether for approximating a chance by substituting similar numbers or for calculating the exact chance if the numbers align. Each case will be provided with three paragraphs — The formula in the first paragraph, an example in the second paragraph, and a general explanation of why it applies (though not necessarily a rigorous proof) in the third paragraph. The total chance will be given on the scale of {{nowrap|0—1}}; focuses are assumed each to have a positive value, as negatives are unintended and a focus with a chance of zero will never get picked by the game's AI meaning they can be excluded from the calculation entirely; and the modifiers applying to the focus' AI will do value (e.g. from AI strategy plans or the bonus for continuing the same branch) have already been applied, as multiplying the result of the rolled dice by a number would be the same as multiplying the ends of a range by that number.<br/>
In shortened form, the ai_will_do value of a focus will be referred simply as the "focus' value".
* {{VisibleCollapse|
1=Let there be <math>n</math> focuses: a focus with the chance of <math>a</math> and <math>n - 1</math> focuses with the chance of <math>b</math>, where <math> a \ge b</math>. In this case, the chance for the focus with the value of <math>a</math> has a chance of <math>\frac{a-b}{a} + \frac{b}{n*a}</math>, while the focuses with the value of <math>b</math> each have the chance of <math>\frac{b}{n*a}</math>.

For example, if there are 3 focuses with values of <math>2</math>, <math>1</math>, and <math>1</math>, the chance for the first focus will be <math>\frac{2-1}{2} + \frac{1}{3*2} = \frac{2}{3}</math>, while the chance for a focus with a smaller chance to be picked is <math>\frac{1}{6}</math> (Since there are 2 such focuses, this adds up to 1 in total).

Intuitively, the chance for the largest-valued focus to get picked consists of 2 mutually exclusive possibilities: the picked value between <math>0</math> and <math>a</math> is larger than <math>b</math> and the opposite: it is smaller. While it's theoretically possible that the same value will get picked for both, {{hover|Due to the way computers store numbers, there is a measurable chance for this occurring, since there is a finite amount of numbers that can be picked. This means that, in practice, it 'almost never' happening is false, using the mathematical definition. However, the calculation will assume that any number from the infinite set of real numbers can be picked.|this [[wp:Almost never|almost never]] happens, so it can be disregarded}}. The chance of the former is the first element of the sum: <math>1 - \frac{b}{a} = \frac{a-b}{a}</math>. If that possibility is fulfilled, the element with the chance of <math>a</math> will get picked every time since other focuses can't get a higher value. If the latter possibility is met, which has the chance of <math>\frac{b}{a}</math>, each focus has an equal chance to be picked. Since there are <math>n</math> focuses in total, the chance gets divided by that number. That is the entire chance for the smaller-valued focuses, while it gets added to the larger-valued focus alongside the chance for the initial possibility.
|2=One focus with a large value and other focuses with the same smaller value}}
* {{VisibleCollapse|
1=Let there be two sequences of focuses: <math>(a_i)_{i=1}^n</math> and <math>(b_i)_{i=1}^k</math> where each focus in a given sequence has the same value — respectively <math>a</math> and <math>b</math>. Focuses are arranged so that <math>a \ge b</math>. The first sequence will have <math>n</math> focuses, while the second sequence will have <math>k</math> focuses. In this case, a focus with the value of <math>a</math> will have the chance of <math>\frac{1}{n} - \frac{kb^n}{n(n+k)a^{n}}</math>, while a focus with the value of <math>b</math> will have the chance of <math>\frac{b^n}{(n+k)a^n}</math>

For example, if there are 3 focuses with a value of 4, and 2 focuses with a value of 3, then the former focuses will each have a chance of <math>\frac{1}{3} - \frac{2 * 3^3}{3 * 5 * 4^3} = \frac{1}{3} - \frac{54}{960} = \frac{160}{480} - \frac{27}{480} = \frac{133}{480} \approx 27.71\%</math>. The latter focuses will each have a chance of <math>\frac{3^3}{5 * 4^3} = \frac{27}{320} \approx 8.44\%</math>

For an understanding of this, it's once again useful to split this into two possibilities: Each focus with the value of <math>a</math> rolled a number between <math>0</math> and <math>b</math> and the reverse: at least one focus has rolled a number above <math>b</math>.<br/>
In regards to the first possibility, this has the chance of <math>\left(\frac{b}{a}\right)^n = \frac{b^n}{a^n}</math>, since this is <math>n</math> independent events, each with a chance of <math>\frac{b}{a}</math>. If this is fulfilled, each individual focus has a chance of <math>\frac{1}{n+k}</math> of being picked, and this is the only way focuses in the sequence <math>b</math> can get picked. As such, this sets the chance for these focuses to be <math>\frac{b^n}{(n+k)a^n}</math>.<br/>
While the chance for the first sequence of focuses can be found by subtracting this – multiplied by <math>k</math> – from <math>1</math> and then dividing the result by <math>n</math>, a different approach will lead to the same result while being more rigorous. For this, we'll take apart the second possibility. If this possibility is indeed the case, then we can safely ignore the focuses with the chance of <math>b</math>. In this case, the chance for a single one of the focuses to be picked is <math>\frac{1}{n}</math>. However, that's taking into account the whole <math>I(0;a)</math> interval, while we need to subtract the <math>I(0;b)</math> interval. The chance for a focus to have the highest value within that interval is <math>\frac{b^n}{n*a^n}</math>, as there are <math>n</math> focuses. As such, the chance that a focus' rolled value falls into the <math>I(b;a)</math> interval and is the highest among focuses is <math>\frac{1}{n} - \frac{b^n}{n*a^n}</math>. Adding the initial value calculated for the first possibility, the resulting formula is <math>\frac{1}{n} + \frac{b^n}{(n+k)a^n} - \frac{b^n}{n*a^n} = \frac{1}{n} - \frac{b^n}{a^n}\left(\frac{1}{n} - \frac{1}{n + k}\right) = \frac{1}{n} - \frac{kb^n}{n(n+k)a^{n}}</math>
|2=Focuses split across two different values}}
* {{VisibleCollapse|
1=Let <math>(a_i)_{i=1}^n</math> be a sequence of focus values. As such, a focus numbered with <math>i</math> will have the value of <math>a_i</math>. Then the chance for the focus <math>i</math> to be picked is <math>\frac{1}{a_i}\int_0^{a_i}\left(\frac{a_i}{x}*\prod_{t=1}^n min(\frac{x}{a_t},1)\right)dx</math>.

For example, with 3 focuses that have values of 1, 2, and 3 respectively, this formula will result in the chances of <math>\int_0^{1}\left(\frac{x}{2}*\frac{x}{3}\right)dx = \int_0^{1}\left(\frac{x^2}{6}\right)dx = \left(\Delta(\frac{x^3}{18})\right)_0^1 = \frac{1}{18} </math> for the first focus, <math>\frac{1}{2}\int_0^{2}\left(min(x,1)*\frac{x}{3}\right)dx = \frac{1}{2}\left(\int_0^{1}\left(\frac{x^2}{3}\right)dx + \int_1^{2}\left(\frac{x}{3}\right) \right) = \frac{1}{2}\left(\left(\Delta(\frac{x^3}{9})\right)_0^1 + \left(\Delta(\frac{x^2}{6})\right)_1^2\right) = \frac{1}{2} \left(\frac{1}{9} + \frac{4}{6} - \frac{1}{6}\right) = \frac{11}{36}</math> for the second focus, and <math>\frac{1}{3}\int_0^{3}\left(min(x,1)*min(\frac{x}{2},1)\right)dx = \frac{1}{3}\left(\int_0^{1}\left(\frac{x^2}{2}\right)dx + \int_1^{2}\left(\frac{x}{2}\right)dx + \int_2^{3} dx\right) = \frac{1}{3}\left(\left(\Delta(\frac{x^3}{6})\right)_0^1 + \left(\Delta(\frac{x^2}{4})\right)_1^2 + \left(\Delta(x)\right)_2^3\right) = \frac{1}{3}\left(\frac{1}{6} + 1 - \frac{1}{4} + 3 - 2\right) = \frac{23}{36}</math> for the third focus.

&nbsp;

In order to calculate this, let's assume that the rolled value for <math>a_i</math> landed on <math>x</math>. If this is the case, the chance for a given different focus with the value of <math>a_t</math> to have a rolled value lower than x is <math>\frac{x}{a_t}</math>. However, since this value can exceed <math>1</math>, the chance should be capped at that amount with <math>min(\frac{x}{a_t}, 1)</math>.<br/>
Since each other focus' generation is an independent event, the chance for all <math>n</math> of them to have a value lower than <math>x</math>, making the focus <math>i</math> picked, is all of these chances multiplied, marked with <math>\prod_{t=1}^n min(\frac{x}{a_t},1)</math>. However, this exact formula includes <math>\frac{x}{a_i}</math> itself, which shouldn't be a factor since this is the focus that is being calculated. In order to cancel this out from the equation, it can be multiplied by <math>\frac{a_i}{x}</math> to produce <math>\frac{a_i}{x}\prod_{t=1}^n min(\frac{x}{a_t},1)</math><br/>
However, this is for a static value of <math>x</math>. In practice, <math>x</math> can take up any value from <math>0</math> to <math>a_i</math>. As such, to get the exact chance, one needs to calculate an infinite sum of infinitely-small values. This is exactly what integrating it does: <math>\int_0^{a_i}\left(\frac{a_i}{x}*\prod_{t=1}^n min(\frac{x}{a_t},1)\right)dx</math>. Though, the scale of this result is unknown instead of being 0–1. The total chance of any focus being picked is <math>1</math> at any given point. Therefore, the total upper bound of the scale will be <math>\int_0^{a_i}1 dx = a_i</math>. In order to find the chance on the scale of 0–1, it'll need to be divided by this result as <math>\frac{1}{a_i}\int_0^{a_i}\left(\frac{a_i}{x}*\prod_{t=1}^n min(\frac{x}{a_t},1)\right)dx</math>, which is the final formula.
|2=General formula}}
=== Other ===

Another important aspect of the focus is <code>cost = 8</code>. This sets how long the focus takes to complete. By default, a cost of 1 is taken to be {{Read define|NFocus|FOCUS_POINT_DAYS|ref=yes|beforeref="points",|spacebeforeref=yes}} of which by default 1 is completed daily, although it's possible to set different speeds depending if the country is at war or at peace.<ref>{{Cite defines|NFocus|FOCUS_PROGRESS_PEACE|FOCUS_PROGRESS_WAR}} in [[Defines]]</ref> In other words, a cost of 1 represents a week by default. Decimals within cost are supported, and it will get rounded down to a whole day in the game.

<code>will_lead_to_war_with = TAG</code> is used to mark the focus as leading to a war towards the specified country. This will show up for the specified country and its allies (subjects, overlord, and/or fellow faction members) as the country doing the focus justifying a wargoal on them in the alerts topbar. This also makes the AI prepare for a potential war, both for the country doing the focus and for the country on whom the focus is set to declare war on.

<code>cancelable = no</code> will set the focus to be impossible to cancel manually without defining a <code>select_effect</code>. This does not prevent the focus from cancelling automatically. However, <code>cancelable = yes</code> ''does not'' make the focus possible to manually cancel if the focus also contains <code>select_effect = { ... }</code>.

== Shared focuses ==
{{SVersion|1.13}}
Shared focuses are those that can be used in multiple national focus trees. Examples of these include the [[Chinese_shared national focus tree branches|"Invite foreign investors" branch]] present in focus trees for {{flag|China}}, {{flag|Communist China}}, and {{flag|Manchukuo}} or the [[Baltic shared national focus branches#Rejoin_the_Railways_Branch|"Rejoin the railways"]] and [[Baltic shared national focus branches#Restore_the_Worker.27s_Republic_Branch|"Restore the worker's republic"]] national focus tree branches for {{flag|Estonia}}, {{flag|Latvia}}, and {{flag|Lithuania}}.

A shared focus is defined with a <code>shared_focus = { ... }</code> block, '''not''' inside of any <code>focus_tree = { ... }</code> block. Within that block, the focus is defined in ''exactly'' the same way as a regular focus within a focus tree: each of the arguments that can be used in a regular <code>focus = { ... }</code> can be used with a <code>shared_focus = { ... }</code> without any changes.

A shared focus is added to a focus tree by adding the <code>shared_focus = my_shared_focus</code> argument within that focus tree. This will add the specified shared focus '''and every shared focus that's connected to it via prerequisites''', assuming that allow_branch is true. Even if the shared focus has a different focus as a prerequisite that's not loaded in this focus tree, it will appear as visible. '''Setting this to a non-existing focus causes a game crash when loading into the main menu.''' In addition, having no non-shared focuses in your tree while your shared focus is added will result in a crash.

In addition, the [[Effect#load_focus_tree|load_focus_tree effect]] can be used with kept completed focuses as such in order to have shared focuses be kept as complete when switching to a different tree:
<pre>load_focus_tree = {
    tree = my_focus_tree
    keep_completed = yes
}</pre>
This can serve as an alternative to using <code>allow_branch = { ... }</code> in order to swap out focus tree branches for the same country from being visible or not.

Shared focuses can be defined in any {{path|common/national_focus/*.txt}} file. Usually they're kept in different files from the focus trees using them, but that's not necessary.

=== Joint focuses ===
{{SVersion|1.13}}
Joint focuses are a subset of shared focuses which also share the completion of focuses between different countries. The effects may apply to the completer, only other countries, or all of them.<ref>[[forum:1598797|Developer Diary {{!}} Modding!]], [[Paradox]] [[developer diary]] for [[1.13]].</ref>

A joint focus is defined as a root-level <code>joint_focus = { ... }</code> block. Each attribute possible to use in regular national focuses is also possible to use here, with extra additions and changes:
* <code>joint_trigger = { ... }</code> is a trigger block that determines which countries are considered joint for this particular focus. This is checked for every country. Completion will be shared between all coun
* <code>completion_reward = { ... }</code> functions similarly to regular national focuses, but executes the effects on every country which is joint.
* <code>completion_reward_joint_originator = { ... }</code> is a completion reward executed only for the country that has directly completed the focus.
* <code>completion_reward_joint_member = { ... }</code> is a completion reward executed to every country where the focus is completed other than the one that has directly completed it.
Other than these, joint focuses are identical to shared focuses.

== Continuous focuses ==
{{SVersion|1.13}}
Continuous focuses are defined in {{path|common/continuous_focus/*.txt}} files. These are the focuses that can be selected after having completed {{rd|NCountry|MIN_FOCUSES_FOR_CONTINUOUS|ref=yes|beforeref=national focuses|spacebeforeref=yes}} and which last eternally without being able to complete them entirely.
=== Focus palettes ===
A continuous focus palette is incredibly similar to a definition to a national focus tree, but with less arguments.

<code>id = my_focus_palette</code> decides the ID of the focus palette. Mandatory.

<code>country = { ... }</code> is a [[AI modding#AI will do|MTTH block]] that assigns scores to each country for picking a continuous focus palette, in the exact same manner as it's done with national focus trees.

<code>default = yes</code> assigns a focus palette as default. As with national focus trees, only one palette should be default total: no more, no less. Defaults to false.

<code>reset_on_civilwar = no</code> similarly as with national focuses, has an unknown effect.

<code>position = { x = 100 y = 1230 }</code> assigns the default position of the continuous focus palette, measuring in the pixel position of the top left corner. This will get used if the national focus tree of the country doesn't specify a different position that is not <code>x = 0 y = 0</code>

<code>focus = { ... }</code> are the continuous focuses themselves.

Example: {{VisibleCollapse|
<pre>continuous_focus_palette = {
    id = BHR_focus_palette
    country = {
        factor = 0
        modifier = {
            tag = BHR
            add = 10
        }
    }
    position = { x = 50 y = 1000 }
    focus = {
        ...
    }
    focus = {
        ...
    }
}</pre>}}
=== Focuses ===
Continuous focuses are also fairly similar to national focuses in definition, but there are substantial differences.

<code>id = TAG_focus_name</code> is the continuous focus' ID. There must be no overlap across different palettes. This also gets used as the localisation key for the focus' name and appending <code>_desc</code> is used to get the focus' description.

<code>icon = GFX_focus_icon_name</code> is the sprite of the icon used by the continuous focus. Similarly to national focus, there must be both a regular icon and a shine for it to always work properly.

<code>available = { ... }</code> is a trigger block required to be met for the focus to be ''visible'', unlike national focuses where this makes it possible to pick. Instead, the trigger block used for making the focus be possible to pick is <code>enable = { ... }</code>. <code>available_if_capitulated = yes</code> will also make the focus possible to select while capitulated. If false, the focus will still be visible, but can't be picked.

<code>modifier = { ... }</code> is a modifier block that details the list of [[modifiers]] and their values that are added by having the focus selected. However, there are other ways to apply continuous bonuses aside from modifiers, such as research bonuses for specific categories and equipment bonuses. For these, <code>idea = idea_name</code> can be used, which'll add the idea to the country when the focus is selected and remove it when the focus is cancelled. This is recommended to do with hidden ideas, as it shows the effects of the idea when hovering over the focus.

<code>select_effect = { ... }</code> and <code>cancel_effect = { ... }</code> are effects executed when selecting and unselecting this focus respectively. Since it's possible to select and unselect the focus at any time with no cost, it's recommended to make these be completely the opposite of each other so that doing so will not grant the player any benefit.

<code>daily_cost = 0.4</code> is the cost in daily political power gain to take this focus compared to not having any focus selected, can be decimal.

<code>ai_will_do = { ... }</code> is a [[AI modding#AI will do|MTTH block]], working in the exact same way as [[#AI will do|ai_will_do in national focuses]]. Additionally, <code>supports_ai_strategy = AI_focus</code> makes the focus be possible to pick by AI if [[AI focuses|it is following the specified focus]]. Without that, AI would never pick this continuous focus.

        has_war = no
    }

    supports_ai_strategy = ai_focus_peaceful

    modifier = {
        stability_weekly = 0.005
        war_support_weekly = -0.001
    }
}</pre>}}{{VisibleCollapse|
<pre>focus = {
    id = BHR_focus_name_2
    icon = GFX_focus_icon_name_2
    daily_cost = 1.2

    available = {
        has_war = yes
    }
    enable = {
        surrender_progress > 0.5
    }

    ai_will_do = {
        base = 4
        modifier = {
            factor = 2

}</pre>}}

== AI strategy plans ==
{{Main|AI modding|AI Strategy Plans|section=1}}
{{SVersion|1.13}}
AI strategy plans to tell AI what to prioritise depending on circumstances: which advisors to pick, which technologies to research, which AI strategies to apply, and, most importantly here, which focuses to pick. This is a short overview of AI strategy plans purely for national focus prioritising, full detail being in the [[AI modding]] article. Several AI strategy plans can be enabled at the same time.

AI strategy plans are defined within {{path|common/ai_strategy_plans/*.txt}} files. Within these files, a new strategy plan is done as a new block, the name of which must be the same as the internal ID of the plan.

Within that plan, <code>name = "AI plan's name"</code> and <code>desc = "AI plan's description"</code> decide the name and the description of the strategy plan. ''This is never intended to be shown to the player'', so localising it into different languages is never needed. Instead, this is used within the <code>aiview</code> console command, which tells info to the developer about what AI wants to prioritise.

<code>allowed = { ... }</code> is, similarly to decisions or ideas, is a trigger block '''checked only at the game's start'''. This is primarily used to tell which country and DLCs to restrict the strategy plan to.

<code>enable = { ... }</code> is checked each day if the allowed is met. If <code>enable = { ... }</code> is met, then the AI strategy plan will be assigned to the AI regardless of whether <code>enable = { ... }</code> turns false later or not. Commonly, [[Triggers#has_game_rule|the has_game_rule trigger]] is used to make it work with custom game rules deciding what path the AI will pick. [[Triggers#is_historical_focus_on|is_historical_focus_on]] is commonly used with the default AI game rule, and country flags can be used for randomisation, by setting up an [[On actions|on_startup]] to set a random one using [[Scopes#random_list|random_list]]

<code>abort = { ... }</code> is checked every day in order to make the AI ''stop'' using this AI strategy plan if <code>enable = { ... }</code> is met. Additionally, it must be false in order for the AI strategy plan to be possible to be picked.

<code>ai_national_focuses = { TAG_focus_name_1 TAG_focus_name_2 }</code> is a list of national focuses, separated by whitespaces, in the order that the AI should take them. In this example, the AI will try to take TAG_focus_name_1 first if possible. If it's already taken or TAG_focus_name_1 is impossible to take, then AI will try to take TAG_focus_name_2. If both of the focuses are impossible to take due to being completed or unavailable, then it will move on to other focuses, taking <code>ai_will_do = { ... }</code> into consideration. While following a focus order, it ignores <code>ai_will_do = { ... }</code> values.

<code>focus_factors = { ... }</code> assigns a multiplier to ai_will_do values of the specified focus. An entry in this block looks like <code>TAG_focus_name = 3</code>. In this case, this will make the ai_will_do value of the focus be multiplied by 3, assuming AI strategy plan's weight of 1. If the focus has an ai_will_do value of 4 after applying modifiers, it'll become 12 if AI is following this strategy plan, and get treated as such. And, of course, a factor of 0 will make the focus be never picked without specification in ai_national_focuses. This can serve as a faster-to-write or a more randomised way to make AI follow a political path by making focuses it should never pick have a value of 0.

<code>weight = { ... }</code> is a [[AI modding#AI will do|MTTH block]] assigning an overall weight to the plan. Defined in the same way as [[#AI will do|a national focus' ai_will_do]], this multiplies each factor within the AI strategy plan by the weight before applying them. A weight of 1.25 will turn a focus factor of 4 into 5 before applying it, for instance. This can be used to make the AI follow the strategy plan more strictly in some cases and less strictly in others.

== Notes ==
{{cnote|a|Dynamic countries, when created, will go through the check assigning a focus tree again. As well as that, reloading focuses by saving over a focus tree file while the debug mode is turned on via launch settings will also refresh this check.}}
{{cnote|b|The default position is defined within the pallete's definition in a {{path|common/continuous_focus/*.txt}} file}}
{{cnote|c|The exact size of a single x and y coordinate unit uses the focus_spacing positionType within {{path|interface/nationalfocusview.gui}}}}
{{cnote|d|If [[Effect#mark_focus_tree_layout_dirty|mark_focus_tree_layout_dirty]] is put within a focus reward, it wouldn't be marked as complete for the [[Triggers#has_completed_focus|has_completed_focus]] trigger at the time it's executed. This can be avoided by using a hidden event, fired immediately, which has the effect to refresh the check within its immediate. Alternatively, the [[Effect#load_focus_tree|load_focus_tree]] effect, set to have the focuses kept complete, can be used to mark the focus as complete before doing the refreshing.}}
{{cnote|e|2=Within folder paths, a backslash (\) can result in the game not being able to read the folder, since it's expected to be used as an [[wp:Escape character|escape character]]. Instead, it's preferable to use forward slashes, as in <code>texturefile = gfx/interface/goals/my_file.dds</code>.}}
